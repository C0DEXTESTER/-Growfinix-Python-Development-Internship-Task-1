# CLI Log Parser for Tour Enquiries

## Files
- `log_parser.py` — the CLI tool (argparse + re)
- `sample_enquiries.txt` — a realistic, messy sample log (58 blocks: mixed
  formats, missing fields, noise/junk lines, unlisted destinations)
- `generate_sample.py` — the script that generated the sample file (test-data
  only, not part of the actual tool)

## How it works
1. **Read** the raw log file (`argparse` handles the file path + flags).
2. **Split** the text into enquiry "blocks" (separated by blank lines),
   filtering out obvious noise lines (headers, `[SYSTEM]`/`[WARN]` tags,
   separators like `----`/`====`).
3. **Extract** per block using `re`:
   - **Email** — a standard `user@domain.tld` pattern.
   - **Destination** — matched against a known destination list
     (`KNOWN_DESTINATIONS`); anything not on the list is reported as
     `"Unknown"` rather than silently dropped.
   - **Name** — tried against several common phrasings seen in real
     enquiries (`"I'm ..."`, `"Name: ..."`, `"From: ..."`, `"submitted by
     ..."`, `"Name (email)"`, etc.), in priority order.
4. **Print** a clean `Name | Email | Destination` table straight to the
   terminal, plus a summary (total scanned, parsed, unparsable, and a
   count of enquiries per destination).

Missing fields are shown as `N/A`; blocks with **no name and no email at
all** are treated as noise/junk and excluded from the table (visible via
`--verbose`).

## Usage
```bash
# Basic run — prints the clean table + summary
python3 log_parser.py sample_enquiries.txt

# Only show entries whose destination didn't match the known list
python3 log_parser.py sample_enquiries.txt --unknown-only

# Also save the parsed results to a CSV file
python3 log_parser.py sample_enquiries.txt --export results.csv

# Show the raw text of blocks that couldn't be parsed at all
python3 log_parser.py sample_enquiries.txt --verbose

# Built-in help (auto-generated by argparse)
python3 log_parser.py --help
```

## Design notes / gotchas worth knowing for your submission
- **Why `[ \t]+` instead of `\s+` inside the name pattern?** `\s` also
  matches newlines. Using `\s+` let a captured name accidentally swallow
  the next line's label (e.g. `Name: Aisha Smith\nEmail:` became
  `"Aisha Smith Email"`). Restricting to spaces/tabs keeps a name within
  a single line.
- **Why not just use `re.IGNORECASE` everywhere?** For trigger phrases
  like `"hi im"` case-insensitivity is fine and needed. But turning it on
  for the *whole* pattern also makes `[A-Z]` match lowercase letters,
  which let filler words like `"and"` get captured as part of a name
  (`"my name is Grace Johnson and my email..."` → wrongly captured
  `"Grace Johnson and"`). The fix: scope the case-insensitive flag to
  only the trigger phrase using `(?i:...)`, and keep the actual name
  capture case-sensitive.
- **Destinations not in `KNOWN_DESTINATIONS`** are reported as `"Unknown"`
  rather than guessed at — guessing arbitrary destination names from free
  text reliably is much more error-prone than name/email extraction, so
  this tool sticks to a known-list match as you specified. Add more
  cities/countries to the `KNOWN_DESTINATIONS` list as needed.

## Extending it
- Add destinations: just add strings to `KNOWN_DESTINATIONS` at the top
  of `log_parser.py`.
- Add name phrasings: add a new pattern to `NAME_PATTERNS`, following the
  same `[ \t]+` / scoped-case-insensitivity conventions above.
- Want CSV always saved, not just on-demand? Call `export_csv()`
  unconditionally in `main()`.
